发表于 2016-10-14 | 分类于 Android |
在研读了RxJava的框架源码后，感悟较深。在工作上与团队同事做出PPT分享的同时，也在此记录总结。当然，对于框架学问来说，尤其是RxJava的模式思想，这只是第一弹。

响应式编程思维

What

本质上就是使用异步数据流（streams）进行编程。在此基础上，可以对这些数据流进行创建、转换、过滤、映射等操作。可参考stackoverflow上的一个回答：What is (functional) reactive programming?

RxJava is a Java VM implementation of Reactive Extensions: a library for composing asynchronous and event-based programs by using observable sequences.

官方定义：RxJava是Reactive Extensions的Java VM实现：用于通过使用observable序列来组合异步和基于事件的程序的库。
所以 observable 可认为是 streams的实现。

Why

  • 清晰的业务流程
    只需要关注业务逻辑相互依赖的事件，及数据流。不需要关心具体实现。
  • 便于业务的维护和扩展
    不需要改变原有数据流内部，采用增删中间逻辑处理单元(数据流)来组合的方式即可。
  • 便于调试和测试
    中间逻辑处理单元的细化，可帮助快速定位有利于调试
    Mock业务流程数据流的逻辑，即可进行测试
  • 代码简洁易读，无缩进
    建议使用方法细化的方式，尽量少采取内部类内部方法的方式实现数据流

原理

Action、Function


可以这么理解：
Action：用于事件，即某个数据流的动作
Function：用于数据流，可做中间处理环节
本质都是 call()接口回调方法
区别在于，Function是有返回值的，而Action没有

Observable

可观察的对象（被观察者），本质上是数据流(stream)，可发射基于时间排序的事件序列。
重要属性： OnSubscribe onSubscribe
本质定义：

实际作用：call方法里进行事件的发射

Subscriber

订阅者(观察的对象)，本质上是提供数据流的回调,与数据流的关联。
回调生命周期：

  1. onNext -> … -> onNext -> onCompleted
  2. onNext -> … -> onNext -> onError

Subscription

Observable 与Subscriber 的关联
subscription= observable.subscribe(subscriber)
tip：可用于终止数据流发射事件序列，一般在程序生命周期内调用subscriber.unsubscribe()可避免内存泄露

Producer　(backpressure)

可以这么理解：允许subscriber向上流Observable要求发射一定数量item的请求通道，即控件上流的输入和下流的输出
一般实现会extends AtomicLong,本身就是计数器。且内部有两个方法：request(long),produce(long)

源码定义

What?!!
参数是Subscriber，返回结果也是Subscriber？
而且泛型定义Operator是不是搞错了？
不应该是Operator么？

其实呢，暂时可以这么简单地理解：操作符就是数据流方向中间处理单元，这个处理操作本身会降级（Observable ->Subscription），源码内部会做升级（Subscription -> Observable）,所以从源码定义来看，操作符可以看做是上下流的订阅者。
